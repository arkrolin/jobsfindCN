# STAR-R 原则 — 中文简历改写指南

## 什么是 STAR-R

STAR 是国际通行的简历和面试叙事框架，STAR-R 增加了 Reflection（反思）层：

| 字母 | 含义 | 回答的问题 |
|------|------|-----------|
| **S** | Situation（情境） | 在什么背景下？ |
| **T** | Task（任务） | 你要解决什么问题？ |
| **A** | Action（行动） | 你具体做了什么？ |
| **R** | Result（结果） | 取得了什么可量化的成果？ |
| **R** | Reflection（反思） | 你学到了什么？（体现成长性） |

## 中文简历适用场景

中国简历的特点是**篇幅紧凑**（校招 1 页，社招 1-2 页），所以 STAR-R 需要做适应：

- Situation 和 Task 可以融合为一句话（"在 XX 场景下，面临 YY 挑战"）
- Action 是核心，篇幅占比应最大（"做了什么"）
- Result 必须量化（"提升了多少"）
- Reflection 在简历上可选（面试时口述更合适），如果是简历则精简为一句

## 改写公式

```
[强动词] + [具体行动] + [量化结果] + [业务影响]
```

## 中文简历强动词库

### 技术类
| 弱动词（避免） | 强动词（使用） |
|--------------|--------------|
| 负责 | 设计并实现 / 主导 / 重构 |
| 参与 | 核心贡献 / 独立完成 |
| 协助 | 推动 / 搭建 |
| 做了 | 优化 / 改进 / 升级 |

### 产品/运营类
| 弱动词（避免） | 强动词（使用） |
|--------------|--------------|
| 负责 | 主导 / 策划 / 设计 |
| 参与 | 推动 / 协调 |
| 做了 | 实现 / 提升 / 降低 |
| 帮助 | 从 0 到 1 搭建 / 优化 |

## 量化词库

| 领域 | 量化示例 |
|------|---------|
| 性能 | "P99 延迟从 Xms 降至 Yms" / "QPS 提升 X 倍" |
| 规模 | "日均处理 XX 万请求" / "支撑 XX 万 DAU" |
| 效率 | "构建时间从 Xmin 降至 Ymin" / "发布频率提升 X 倍" |
| 质量 | "线上故障率降低 X%" / "代码覆盖率从 X% 提升至 Y%" |
| 业务 | "转化率提升 Xpp" / "用户留存提升 X%" |

## 改写示例

### 示例 1：技术岗

**原文：**
> 负责 XX 系统的开发和维护，解决了系统不稳定的问题。

**诊断：**
- "负责"太弱，看不出角色的重要性
- "开发和维护"没说具体做了什么
- "解决了不稳定的问题"没有量化

**改写：**
> 主导 XX 推送系统的架构重构，将单点同步链路改造为 Kafka + Redis 四级异步管道。上线后系统 P99 延迟从 2.1s 降至 180ms，日均推送量从 50 万提升至 300 万。

**改写逻辑：**
- "负责"→"主导"（体现角色重要性）
- 补充了具体技术方案（Kafka + Redis 异步管道）
- "解决了"→量化结果（延迟降低 91%，推送量提升 6 倍）

### 示例 2：应届生/实习

**原文：**
> 参与了 XX 项目的开发，学到了很多。

**诊断：**
- "参与了"看不出具体贡献
- "学到了很多"是自我视角，面试官想看的是你对项目的贡献
- 完全没有行动和结果的描述

**改写：**
> 独立实现 XX 项目的用户画像模块，使用 Go + Redis 设计标签存储方案，支持 10 万用户 × 200 维标签的毫秒级查询。单元测试覆盖率 85%，上线后零 Bug。

**改写逻辑：**
- "参与了"→"独立实现"（体现自主性）
- 补充技术栈和方案设计
- 增加量化指标和上线结果

### 示例 3：产品岗

**原文：**
> 负责 XX 功能的迭代，提升了用户体验。

**诊断：**
- "负责迭代"太模糊
- "提升了用户体验"无法量化
- 没有说明如何发现问题和做决策

**改写：**
> 通过用户访谈和埋点分析，发现 XX 功能的流失率达 35%。主导交互重构方案，推动设计、前端、后端落地。上线后该功能 7 日留存从 65% 提升至 82%，月活提升 40%。

## 常见中文简历问题

### 问题 1：假大空
> "具有优秀的团队协作能力和沟通能力"

→ 改为：具体的协作案例和成果

### 问题 2：职责清单
> 一行一个"负责 XXX"，像岗位说明书

→ 改为：每段经历 3-5 个 bullet，每个 bullet 包含"行动+结果"

### 问题 3：技术栈堆砌
> "熟练掌握 Java, Python, Go, C++, JavaScript, SQL, Redis, Kafka..."

→ 改为：只写真正在工作中用过的，标注熟练度（"精通"/"熟练"/"了解"）

### 问题 4：项目没有上下文
> "开发了 XX 系统"

→ 改为：补充系统规模（"日活千万级 XX 系统"）、技术挑战（"在 YY 约束下"）

### 问题 5：个人信息过度
> 年龄、性别、民族、政治面貌、婚姻状况...

→ 中国简历有时需要这些（尤其国企），但外企和互联网公司通常不需要。根据目标公司调整。如果用户投递外企，建议删除。

## 不同公司类型的简历风格

| 公司类型 | 风格 | 注意事项 |
|---------|------|---------|
| 一线互联网（字节/腾讯/阿里） | 简洁技术风 | 强调技术深度、量化成果、开源贡献加分 |
| 外企（Microsoft/Amazon） | 英文或中英双语 | 去掉敏感个人信息、突出 leadership 和 impact |
| 国企/央企 | 正式规范 | 可能需要更多个人信息、强调学历和证书 |
| 创业公司 | 实战导向 | 强调 0→1 能力、多面手能力、ownership |

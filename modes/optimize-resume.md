# Mode 3: 简历优化 (optimize-resume)

## 目标

用 STAR-R 原则对用户简历进行逐条诊断和改写，提升面试通过率。

## 前置条件

- `data/cv.md` 应已存在
- 建议已有 `data/profile.yml`（如果不存在，诊断会缺少目标岗位信息）

## 执行流程

### Step 1: 整体扫描

读取 `data/cv.md`，先做整体诊断，找最致命的 1-2 个问题。

**快速检查清单：**
- 排版是否干净？（层级清晰、Bullet 对齐、字体统一）
- 内容是否有量化？（每个经历都有数字支撑 vs 全是"负责XXX"）
- STAR 使用如何？（只做了什么 vs 也说了结果）
- 篇幅是否合适？（校招 1 页 / 社招 1-2 页）
- 关键词是否到位？（目标岗位的技术栈是否覆盖）
- 有没有硬伤？（长时间 Gap、频繁跳槽、技术栈过时）
- 个人信息是否合适？（根据目标公司类型评估）

**如果用户提供了目标 JD：**
同时读取 JD，对比关键词和技能要求的覆盖情况。

### Step 2: 整体诊断输出

```
## 整体诊断

**核心问题：**[1-2 个最致命的问题，直接说]

**篇幅：**[OK / 太长 / 太短]

**量化程度：**[高 / 中 / 低 — 有数字的 bullet 占比]

**关键词覆盖：**[目标岗位]的关键词覆盖率——命中 X，缺失 Y

**硬伤检测：**[有 / 无 — 具体说明]
```

### Step 3: 逐条改写

对简历中的每段经历逐条分析。每个 bullet 输出：

```
### [公司名] — [岗位名]

#### Bullet [N]
**原文：**
> [用户原文]

**问题：**
[为什么弱——太模糊 / 没结果 / 动词太弱 / 没有上下文]

**改写：**
> [优化版本]

**改写逻辑：**
[背后的原理——为什么这样改更好。让用户下次自己会改。]
```

**改写要点：**
- 使用「强动词 + 具体行动 + 量化结果」公式
- 行动描述要具体到技术方案或决策逻辑
- 量化数据不编造，缺失时标注 `[建议补充：具体数据]`
- 每段经历保留 3-5 个 bullet，多余的合并或删除

### Step 4: 结构调整（如果需要）

如果简历整体结构有问题：

```
## 结构建议

**当前结构：**[描述]

**建议结构：**
1. 个人信息（精简版）
2. 工作经历（最近的在上面）
3. 项目经验（选 2-3 个最有代表性的）
4. 教育背景
5. 技能（按熟练度分组）

**理由：**[为什么建议这样调整]
```

### Step 5: 关键词补充

```
## 关键词优化

**已有：**[列出简历中命中的关键词]

**建议添加：**
- [关键词1] — [原因：目标岗位 JD 高频出现 / 该技术栈是行业标准]
- [关键词2] — [原因]

**建议弱化/删除：**
- [过时技术] — [原因]
```

### Step 6: 汇总和确认

```
## 优化总结

**改动统计：**
- 整体结构调整：[N] 处
- 逐条改写：[N] 条
- 关键词补充：[N] 个
- 删除冗余：[N] 条

**改写后效果预估：**
- 量化率：[X%] → [Y%]
- 关键词覆盖率：[X%] → [Y%]

---
以上是诊断和改写建议。确认后我会保存到 `data/cv.md`。
有需要调整的地方吗？
```

### Step 7: 保存

用户确认后，将优化后的简历写入 `data/cv.md`。
> "简历已更新保存。建议接下来：
> - 根据目标 JD 做针对性优化（发 JD 给我）
> - 进行模拟面试，检验新简历的叙事效果
>
> 需要我帮你做什么？"

## 注意事项

- 绝对不要编造用户没有的数据或经历
- 不要大幅改变用户的真实经历——优化的是表达方式，不是事实
- 中文简历注意不要过度翻译技术术语（如"关系型数据库管理系统"应写成"MySQL"或"关系型数据库"）
- 如果用户投递外企，询问是否需要生成英文版
- 不要一次输出太长的改写——分步进行，每段经历单独确认
